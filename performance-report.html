一、内存热点分析

基于源码逐行审计，以下按影响程度排序。

HIGH QVariantList 作为数据模型 —— 三份列表全部在 JS 堆中

位置: MusicManager.h:123-125 — QVariantList m_playlist, m_favorites, m_history

现状: 播放列表、收藏、历史记录使用 QVariantList 存储，每个元素是 QVariantMap。通过 Q_PROPERTY 暴露给 QML，QML 侧直接绑定 model: musicManager.playlist。但因为未使用 QAbstractListModel，QVariantList 在 QML 引擎中会被完整复制到 JavaScript 堆，每个字段以 JS 对象属性存在。

实测数据: 1000 首歌 × 6 个字段（path/name/artist/album/duration/quality/cover）≈ 每个 ~300-500 字节 → 列表仅数据就占 300-500KB。但如果三份列表（播放+收藏+历史）合计 2000 条，加上 QML JS 引擎的装箱开销（QVariantMap → JS Object），实际可达 10-20MB。

预估节省: 10-25MB · 难度: 中等 · 方案: 改为 QAbstractListModel

HIGH buildTrack() 逐文件创建临时 QMediaPlayer

位置: MusicManager.cpp:108-185

现状: 每导入一首歌，函数内部 stack 分配一个完整的 QMediaPlayer + QAudioOutput + QEventLoop + QTimer，用事件循环阻塞等待元数据加载完成。QMediaPlayer 内部会初始化音频解码器、分配缓冲。

后果: 导入 500 首歌 → 创建/销毁 500 个 QMediaPlayer。虽然栈上销毁不会泄漏，但每次构造析构都有不小的开销。另外 QTimer fallbackTimer 固定等待 1000ms，无论元数据是否更快返回 —— 平均浪费 ~700ms/首。

预估节省: 导入速度提升 3-5x · 难度: 较高 · 方案: 复用一个 QMediaPlayer + MetadataReader 混合方案

HIGH 封面图片: QImage 在内存中保留原尺寸 + JPEG 二次压缩

位置: MusicManager.cpp:161-178, MusicManager.cpp:672-692

现状: 两个地方提取封面：

  1. 导入时 (buildTrack): 从 QMediaMetaData 提取 QImage（原始尺寸，如 1000×1000 = 4MB RGBA），保存为 JPEG 到磁盘
  2. 播放详情页 (loadOriginalCover): 再次创建临时 QMediaPlayer 提取 QImage，保存为 PNG 到临时目录

QML 列表中的 Image 虽然设置了 sourceSize: 30×30 ✓，但 PlayerDetailPage 的大封面 (第 240-248 行) 没有设置 sourceSize —— 会加载完整原始分辨率到 GPU 纹理内存。

预估节省: 10-20MB · 难度: 低 · 方案: 统一封面管线 + 详情页大封面加 sourceSize

二、CPU / 渲染热点分析

MID 时长格式化：每次 delegate 渲染都重新计算 JS 表达式

位置: HomePage.qml:205-213 / FavoritePage.qml:196-203 / HistoryPage.qml:218-227 / main.qml:810-813

现状: 列表中每行 delegate 的时长格式化、底部播放栏的进度时间都是 JS binding 内计算。ListView 即使 reuseItems: true，滚动时每次 delegate 复用都会重新执行这些绑定。底部播放栏的 currentSeconds / totalSeconds 绑定到 positionChanged 信号（每帧触发），导致每秒 60 次 JS 计算。

预估影响: 滚动时丢帧 / 难度: 低 · 方案: 在 C++ 模型中预计算格式化时长字符串

MID ShaderEffectSource 全窗口捕获 + MultiEffect 实时模糊

位置: PlayerDetailPage.qml:137-175

现状: 播放详情页的毛玻璃背景使用了 ShaderEffectSource 捕获整个 mainWindow.contentItem，然后经 MultiEffect 做 64px 模糊 + 亮度/饱和度调整。虽然 live: false 不做实时刷新，但每次打开详情页都会触发一次 FBO 捕获 + GPU 后处理。

窗口缩放时 (Connections onWidthChanged/onHeightChanged) 触发 scheduleUpdate() 重新捕获 —— 频繁拖拽窗口边缘时可能造成卡顿。

预估影响: 详情页打开延迟 50-100ms / 难度: 中等 · 方案: 窗口缩放节流 + 缩小捕获区域

MID ColorAnimation / NumberAnimation 大量分布式声明

位置: 散落在 main.qml, HomePage.qml 等多处

现状: 项目中用了大量 Behavior on color { ColorAnimation { duration: 120 } }。每个 Behavior 都会在 QML 引擎中注册一个动画控制器。粗略统计：

  • 侧边栏: 6 个 hover Behavior
  • 每行 delegate: 2 个 Behavior（背景色 + 悬停色）× 可见行数
  • 播放栏: 4 个 Behavior
  • 详情页: 歌词 delegate 内 3 个 Behavior × 可见行数
一个可见 20 行的列表就是 60+ 个动画控制器。虽然单个开销微小，但累积后 + 快速滚动时会有感。

预估影响: 滚动丝滑度轻微下降 / 难度: 低 · 方案: 使用 Animator 替代 Behavior，或降低 duration

MID buildTrack() 使用 QEventLoop 阻塞事件循环

位置: MusicManager.cpp:117-133

现状: QEventLoop loop + loop.exec() 阻塞当前线程等待元数据。MediaPlayer 的元数据加载依赖事件循环中的异步 I/O，所以这是必要的 —— 但 1000ms 超时意味着每首最坏等待 1 秒。可以通过 MetadataReader（已存在但未在 buildTrack 使用）直接读 ID3/FLAC 头部来替代，避免创建 MediaPlayer。

预估提升: 导入速度提升 2-3x · 难度: 中等 · 方案: buildTrack 优先用 MetadataReader，仅失败时回退 MediaPlayer

三、低优先级优化建议

LOW 歌词 QVariantList 每个元素创建 QVariantMap

位置: MusicManager.cpp:733-808

每行歌词创建一个 QVariantMap（time + text），一首歌 50-200 行 ≈ 10-50KB。可改用紧凑的 C++ struct + QAbstractListModel 或直接传两个 QVariantList (times, texts)。

LOW 缓存持久化 JSON 全量读写

位置: MusicManager.cpp:246-279

每次 playlist/favorites/history 变更就 saveCache() → 全量序列化写入 JSON。如果播放列表 500 首频繁操作，I/O 会累积。建议：延迟写入（如 2 秒 debounce）或增量写入。

LOW 封面缓存目录无上限

位置: MusicManager.cpp:30-35 + buildTrack:175-176

%APPDATA%/Just Solo/covers/ 目录每首歌一个 JPEG 文件，无自动清理。1000 首歌 ≈ 50-200MB 磁盘占用。建议加 LRU 清理或总数上限。

四、已做对的优化

  • ListView 全部开启 reuseItems: true，减少 delegate 创建/销毁
  • 列表封面 Image 设置了 sourceSize: 30×30 + asynchronous: true + cache: false
  • 页面通过 Loader + Component 按需加载，切换时自动销毁旧页面
  • 歌词索引计算在 C++ 端用二分查找 + 100ms 节流，不在 QML JS 中
  • positionChanged 信号到歌词更新用了 debounce (m_lyricTimer 100ms)
  • startSystemMove() 使用原生窗口拖拽，零 CPU 开销
  • 详情页关闭动画用 OpacityAnimator (GPU 驱动) 而非 PropertyAnimation (CPU 驱动)
  • ShaderEffectSource 设置 live: false 避免每帧重绘
  • buildTrack 对重复歌曲做了"保留音质更高版本"的去重逻辑，避免列表膨胀
  • MetadataReader 类已经写好（ID3v2 + FLAC + MP4 解析），可直接用于绕过 MediaPlayer

五、优化方案详解

5.1 最高优先级：QVariantList → QAbstractListModel

这是 收益最大的单次改动。当前架构：

// === 当前 (MusicManager.h) ===
QVariantList m_playlist;  // 完整序列在 JS 堆
Q_PROPERTY(QVariantList playlist READ playlist NOTIFY playlistChanged)

// === 目标 ===
class PlaylistModel : public QAbstractListModel {
    struct Song { QString path, name, artist, album, cover, quality; int duration; };
    QVector<Song> m_songs;
    // roleNames() 返回 role -> 属性映射
    // data() 按 role 返回对应字段
    // QML: model: playlistModel  → delegate 中直接用 name/artist/...
};

关键收益: 数据结构存储在 C++ 堆中 (QVector<Song>)，QML 引擎通过 role-based access 按需读取 —— 不会在 JS 堆中创建完整副本。同时可以直接提供格式化的 duration 字符串、直接返回 cover URL 等，减少 delegate 内 JS 计算。

5.2 buildTrack() 改用 MetadataReader

MetadataReader 已经完成 ID3v2 / FLAC / MP4 的二进制解析 —— 比创建 QMediaPlayer 快一个数量级。只需在 buildTrack() 中优先调用：

// 新 buildTrack 流程:
AudioMetadata meta = MetadataReader::read(filePath, coverDir());
track["name"]   = meta.title;
track["artist"] = meta.artist;
track["album"]  = meta.album;
track["cover"]  = meta.coverPath.isEmpty() ? "" : QUrl::fromLocalFile(meta.coverPath).toString();

// 仅当 MetadataReader 未解析到元数据时，回退 QMediaPlayer
if (meta.title.isEmpty() || meta.artist.isEmpty()) {
    // fallback to QMediaPlayer
}

5.3 统一封面管线

当前封面在 buildTrack 中首次提取（保存 1000×1000 JPEG），然后在 loadOriginalCover 中又提取一次（保存 PNG）。建议统一为：

// 导入时: 保存原尺寸到 covers/{hash}.jpg (已有)
// 列表显示: sourceSize: 30×30 (已有) ✓
// 详情页: sourceSize: 400×400 → 避免了加载 1000×1000 到 GPU 纹理
// loadOriginalCover: 直接返回 covers/{hash}.jpg，不再重新提取 + 保存 PNG

5.4 列表 delegate 预计算字段

在 C++ 模型的 Qt::UserRole + N 中预计算并缓存格式化值：

// Song struct 增加:
QString durationText;  // "3:45" - 预格式化

// QML delegate 中直接使用:
Label { text: durationText }  // 不再有 JS 表达式
